Disaster Recovery

Business Continuity Planning: RPO/RTO Targets, DR Site Architecture, Runbook Procedures, and Testing Cadence
Version: 3.0
Last Updated: February 2026
Classification: Confidential
Owner: Platform Engineering

1. Overview

Disaster recovery (DR) planning ensures Skolyn can resume operations within defined time objectives following catastrophic events that render the primary deployment site inoperable. Unlike high availability (which addresses component-level failures within a single site), disaster recovery addresses site-level failures caused by natural disasters, facility incidents (fire, flooding, power grid failure), ransomware attacks, or data center decommissioning. As a medical imaging AI platform that processes clinical data and supports time-sensitive diagnostic workflows, Skolyn's DR plan must meet stringent healthcare continuity requirements, including HIPAA business associate agreement obligations for maintaining the availability of electronic protected health information. This document defines recovery objectives, DR architecture, automated failover procedures, data replication strategies, and the quarterly DR testing program that validates recovery capabilities.

Tier 1 components are replicated synchronously to the DR site with sub-5-minute RPO using database streaming replication and real-time object storage mirroring. Tier 2 components use asynchronous replication with hourly snapshots. Tier 3 components are backed up daily to the DR site with daily RPO acceptable for non-time-sensitive archival data.

3. DR Site Architecture

The DR site is a geographically separated deployment (minimum 100 miles from the primary site) that maintains a warm standby of the complete Skolyn platform. The DR site runs the identical software stack with the same model versions, pre-configured DICOM AE titles, and integration endpoints. The DR infrastructure is sized at 100% of primary site capacity for full workload assumption. Key DR site components include: replicated PostgreSQL cluster (streaming replication with 30-second lag), MinIO object storage with bucket-level replication (continuous sync), GPU inference servers pre-loaded with model weights (cold standby — models loaded but not actively serving), and preconfigured network connectivity to customer PACS/RIS/EHR systems via VPN tunnels maintained in standby mode.

4. Failover Procedures

4.1 Automated Failover

DNS-based failover uses Route 53 health checks (or equivalent) monitoring the primary site's health endpoint every 10 seconds. After 3 consecutive failed checks (30 seconds of detected outage), DNS failover automatically redirects the Skolyn service FQDN to the DR site's IP addresses with a TTL of 60 seconds, achieving effective failover within 90-120 seconds. Simultaneously, automated runbooks (implemented as Ansible playbooks triggered by PagerDuty webhooks) promote the DR PostgreSQL replica to primary, activate the MinIO replication endpoint as the primary storage, start GPU inference workers, and activate standby VPN tunnels to customer networks.

4.2 Manual Failover

Manual failover is invoked when the monitoring system detects partial degradation that does not trigger automated failover thresholds but warrants preemptive site migration. The DR manager initiates manual failover via a authenticated CLI command that executes the same runbook as automated failover after human confirmation. Manual failover has a target completion time of 15 minutes from decision to full DR site operational readiness.

5. Data Replication

Data replication employs a multi-layer strategy. Database replication uses PostgreSQL streaming replication with synchronous_commit set to remote_apply, ensuring every committed transaction is confirmed applied at the DR replica before the commit acknowledgment is returned to the application. This provides near-zero RPO for metadata and configuration data. Object storage replication uses MinIO's bucket replication feature with asynchronous mode, achieving average replication lag of 45 seconds under normal load. For studies actively in the inference pipeline at the time of failure, the DICOM receive log (stored in both primary and DR databases) enables automatic re-retrieval of unprocessed studies from the source PACS after DR activation.

6. DR Testing Program

Quarterly DR tests validate the complete failover process, including automated DNS failover, database promotion, GPU inference activation, and end-to-end study processing at the DR site. Each test uses synthetic DICOM studies and simulated PACS connections to avoid impacting production workflows. Test results are documented with measured RTO (target: <1 hour), measured RPO (target: <5 minutes), number of test studies processed, end-to-end latency at DR site, and identified issues with remediation plans. Annual tests additionally include failback procedures (returning operations from DR to primary after primary site restoration).
