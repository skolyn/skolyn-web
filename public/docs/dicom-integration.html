1. Introduction

The Digital Imaging and Communications in Medicine (DICOM) standard is the universal protocol for medical image exchange, and Skolyn's DICOM integration layer is designed to interoperate seamlessly with any DICOM-conformant PACS, modality, or workstation. This document details Skolyn's DICOM conformance statement, supported Service-Object Pair (SOP) classes, transfer syntax negotiation, association management, TLS security configuration, and testing procedures. Skolyn supports both the traditional DICOM Upper Layer Protocol (C-STORE, C-FIND, C-MOVE, C-GET, C-ECHO) and the modern DICOMweb RESTful interfaces (WADO-RS, STOW-RS, QIDO-RS), enabling flexible integration with legacy and modern medical imaging infrastructure alike.

3. Association Configuration

3.1 Association Parameters

Skolyn's DICOM Gateway manages association establishment, presentation context negotiation, and data transfer. The gateway supports up to 64 concurrent associations per instance, configurable via the dicom.concurrent_associations parameter. Each association is authenticated via TLS client certificate verification or AE Title allowlist, depending on the security configuration. Association negotiation follows the standard DICOM Upper Layer Protocol handshake: A-ASSOCIATE-RQ from the SCU, A-ASSOCIATE-AC (or A-ASSOCIATE-RJ) from the SCP, followed by C-STORE DIMSE commands for each instance in the study.

3.2 AE Title Management

Skolyn maintains an AE Title registry that maps remote AE Titles to configured DICOM sources. Only pre-registered AE Titles are accepted for association — unregistered AE Titles receive an A-ASSOCIATE-RJ with a rejection reason of "calling AE not recognized." The AE Title registry is managed via the Management Console (Settings > Integrations > DICOM Sources) or the Skolyn CLI (skolyn dicom ae-title add). Each registered AE Title can be associated with specific module processing rules — for example, studies from AE Title "MRI_SCANNER_01" can be automatically routed to the Rhenium OS (MRI) module only.

4. DICOM Structured Reports

Skolyn generates DICOM Structured Reports (SR) conforming to TID 1500 (Measurement Report) for delivering AI findings back to the PACS. Each SR includes coded finding terms mapped to SNOMED CT (SCT), RadLex, and ICD-10-CM vocabularies. Measurement groups contain numeric measurements (lesion dimensions, volumes, densities) with standardized units (UCUM). The SR is stored as a new series within the original study, preserving the study-series-instance hierarchy. This allows radiologists to review AI findings alongside the source images in their native PACS viewer without any workflow modification.

5.2 TLS Configuration

DICOM TLS is mandatory for all production deployments. Skolyn supports TLS 1.2 and TLS 1.3 for DICOM associations, with TLS 1.3 preferred. The DICOM Gateway presents a server certificate signed by the institution's CA or a publicly trusted CA. Mutual TLS (client certificate verification) is optional but recommended for highest security — when enabled, the PACS must present a valid client certificate during the TLS handshake. Certificate management is handled through the Skolyn Certificate Manager, which supports ACME (Let's Encrypt), manual PEM upload, and integration with enterprise PKI (Microsoft AD CS, HashiCorp Vault PKI).

6. Testing & Validation

6.1 DICOM Echo Test

# Test connectivity using dcm4che echoscu
echoscu -b SKOLYN_AI -c TEST_SCU@10.0.10.50:11112 --tls

# Expected output:
# Received Echo Response (Status: 0000H)
# C-ECHO completed successfully

6.2 C-STORE Test

# Send a test study
storescu -b SKOLYN_AI -c TEST_SCU@10.0.10.50:11112 --tls \
  --tls-protocol TLSv1.3 \
  /path/to/test/study/*.dcm

# Verify study processing
skolyn-cli study status --accession ACC-2026-TEST
