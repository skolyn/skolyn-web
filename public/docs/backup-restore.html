Backup & Restore

1. Overview

Skolyn's backup and restore infrastructure provides comprehensive data protection for all platform components, ensuring the ability to recover from data corruption, accidental deletion, ransomware attacks, hardware failures, and software defects. As a medical imaging AI platform handling protected health information, Skolyn's backup strategy must satisfy HIPAA's data availability requirements, institutional data retention policies (typically 7-10 years for medical imaging data), and contractual SLA obligations. The backup architecture covers four data categories: database metadata (PostgreSQL), DICOM study storage (MinIO object storage), AI model artifacts (model weights, configurations, version metadata), and platform configuration (Kubernetes manifests, secrets, certificates). All backups are encrypted at rest using AES-256-GCM, integrity-verified using SHA-256 checksums, and tested monthly through automated restore validation.

3. Database Backup Strategy

3.1 Continuous WAL Archiving

PostgreSQL Write-Ahead Log (WAL) archiving provides continuous point-in-time recovery (PITR) capability. Every committed transaction is written to WAL segments, which are asynchronously shipped to the backup storage target via pgBackRest. WAL archiving enables recovery to any point in time within the 30-day WAL retention window with transaction-level granularity. This is critical for recovering from logical corruption (e.g., incorrect data modification) where the restore point must be precisely before the corrupting transaction.

3.2 Full Logical Backups

Daily full logical backups (pg_dump with custom format and compression) provide database-level restore capability independent of PostgreSQL version. Logical backups support cross-version restoration (enabling database upgrades), selective table restoration, and schema-only or data-only restoration. Backup jobs run during the maintenance window (02:00-04:00 UTC) and are monitored for completion and integrity. A failed backup triggers an immediate P2 alert with automatic retry.

4. Object Storage Backup

DICOM study data stored in MinIO is protected through cross-region replication. The replication target is a separate storage cluster in a different geographic region, using server-side replication with bandwidth throttling (configurable, default 100 Mbps) to avoid impacting production network paths. Object versioning is enabled, retaining previous versions of overwritten objects for 30 days. For institutions requiring long-term DICOM archive (beyond the online storage retention period), scheduled archival jobs move studies older than the configured threshold (default 24 months) to cold storage (AWS Glacier Deep Archive or Azure Cool Blob Storage) with retrieval times of 12-48 hours.

5. Restore Procedures

5.1 Point-in-Time Recovery

Database PITR restores are performed using pgBackRest, specifying the target recovery timestamp. The restore process: stops the PostgreSQL cluster, restores the most recent full backup prior to the target timestamp, replays WAL segments up to the specified recovery point, and promotes the restored instance. Estimated PITR completion time: 15-45 minutes depending on database size and WAL volume to replay. All restore operations generate comprehensive audit logs documenting the recovery point, operator identity, authorization chain, and data scope affected.

5.2 Full System Recovery

Full system recovery restores the complete Skolyn platform from backups on new infrastructure. The recovery sequence is: provision infrastructure (automated via Terraform/Pulumi), restore platform configuration from Git, deploy Kubernetes manifests, restore PostgreSQL from latest full backup + WAL replay, synchronize DICOM storage from replication target, load AI model artifacts from versioned storage, validate all endpoints via automated health checks, and re-establish DICOM associations with connected PACS systems. Full system recovery target: <4 hours for Tier 1 services.

6. Backup Verification

Monthly automated restore tests verify backup integrity by restoring the most recent backup to an isolated test environment, running automated data integrity checks (row counts, checksum validation, referential integrity), executing a suite of functional tests against the restored platform, and generating a backup verification report documenting test results and any remediation required. Additionally, annual DR exercises include a complete system rebuild from backups on fresh infrastructure to validate the end-to-end recovery pipeline.
