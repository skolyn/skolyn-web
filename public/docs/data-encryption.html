Data Encryption

Cryptographic Data Protection at Rest, in Transit, and in Use for the Skolyn Platform

1. Executive Summary

Data encryption is the cornerstone of Skolyn's data protection strategy. As a medical imaging AI platform processing Protected Health Information (PHI) and special category health data, Skolyn enforces encryption at every stage of the data lifecycle — at rest in storage, in transit across networks, and during active processing in memory. This document describes the cryptographic algorithms, key management architecture, certificate management, and protocol configurations that protect patient data from unauthorized access, tampering, and disclosure. All encryption implementations comply with NIST SP 800-175B, FIPS 140-2 Level 3, and the requirements of HIPAA §164.312(a)(2)(iv) and §164.312(e)(1).

3. Encryption at Rest

3.1 Envelope Encryption

Skolyn employs a multi-layer envelope encryption architecture for all persistent data. Each data object (DICOM file, report, audit record) is encrypted with a unique Data Encryption Key (DEK) generated from a cryptographically secure random number generator (CSPRNG). The DEK is then encrypted (wrapped) by a Key Encryption Key (KEK) that resides exclusively within a Hardware Security Module (HSM). The HSM is FIPS 140-2 Level 3 certified, meaning the KEK never exists in plaintext outside the tamper-resistant hardware boundary.

3.3 DICOM-Specific Encryption

Medical images receive additional protection beyond storage-layer encryption. Skolyn applies DICOM PS3.15 Annex E compliant encryption profiles that encrypt the pixel data, patient demographics, and clinical annotations separately. This enables scenarios where an authorized user can access de-identified images for research while the patient identifiers remain encrypted and inaccessible. The encryption profile uses AES-256-CBC with PKCS#7 padding for DICOM Transfer Syntax compatibility, and the per-study DEK is wrapped with the institutional KEK.

4. Encryption in Transit

4.1 TLS Configuration

All external and internal network communications are encrypted using TLS 1.3 (RFC 8446). The platform enforces the following cipher suite preferences:

# Preferred cipher suites (in order)
TLS_AES_256_GCM_SHA384
TLS_CHACHA20_POLY1305_SHA256
TLS_AES_128_GCM_SHA256

# Deprecated (permanently disabled)
TLS 1.0, TLS 1.1, TLS 1.2 with CBC mode
RC4, DES, 3DES, MD5, SHA-1
Export ciphers, NULL ciphers, anonymous DH

4.2 Mutual TLS (mTLS)

All internal service-to-service communication uses mutual TLS authentication through the service mesh. Each microservice is issued an X.509 certificate with a SPIFFE identity (e.g., spiffe://skolyn.com/ns/production/sa/inference-engine). Certificates have a maximum validity of 24 hours and are automatically rotated by the mesh control plane. The certificate authority (CA) is operated by Skolyn and anchored in the HSM. Certificate chain validation is enforced at every TLS handshake — expired, revoked, or improperly issued certificates are rejected.

4.3 Protocol-Specific Encryption

  • DICOM TLS: DICOM associations use TLS per PS3.15 Annex B. The Association Acceptor validates the Association Requestor's certificate against the configured trust store before accepting the connection.
  • HL7v2 MLLP-TLS: HL7 messaging uses TLS-encrypted MLLP sockets. The healthcare facility's HL7 engine and Skolyn's interface engine authenticate via client certificates.
  • FHIR API: RESTful FHIR endpoints enforce HTTPS-only with HSTS (max-age=31536000, includeSubDomains, preload).
  • DICOMweb: WADO-RS, STOW-RS, and QIDO-RS endpoints use OAuth 2.0 bearer tokens over TLS 1.3.

5. Key Management

5.1 Key Hierarchy

Skolyn's key management architecture follows a three-tier hierarchy designed to minimize the blast radius of any single key compromise:

  • Root Key (Master Key): Generated within the HSM during initial provisioning. Never exported. Used exclusively to generate and protect KEKs. Split across multiple HSM partitions with M-of-N quorum requirements for recovery.
  • Key Encryption Keys (KEK): Generated per-tenant (institution) by the HSM. Stored encrypted in the HSM key store. Used to wrap/unwrap DEKs. Rotated every 90 days with backward compatibility for decrypting historical data.
  • Data Encryption Keys (DEK): Generated per-object using CSPRNG. Wrapped by the tenant's KEK and stored alongside the encrypted object's metadata. Ephemeral in memory — zeroized immediately after encryption/decryption completes.

5.3 Key Destruction

When data is deleted (e.g., GDPR Right to Erasure), Skolyn performs crypto-shredding — the DEK associated with the deleted data is permanently destroyed from the HSM key store. Without the DEK, the encrypted data is computationally irretrievable even if the ciphertext remains in backup archives. A Certificate of Cryptographic Destruction is generated and provided to the Data Controller for compliance records.

6. Encryption Monitoring & Compliance

  • Encryption Coverage Reports: Automated weekly scans verify that 100% of data stores have encryption enabled. Any unencrypted storage is flagged as a P1 security incident.
  • Certificate Monitoring: All certificates are tracked in an inventory with automated alerts at 30, 14, and 7 days before expiration. Expired certificates trigger automated renewal or (if renewal fails) service degradation alerts.
  • HSM Health Monitoring: HSM operational status is continuously monitored with alerts for hardware failures, capacity thresholds, and anomalous usage patterns.
  • Compliance Validation: Quarterly cryptographic assessments verify algorithm compliance with current NIST recommendations and industry standards.
