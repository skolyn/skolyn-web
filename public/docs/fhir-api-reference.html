1. Overview

Skolyn exposes a standards-compliant HL7 FHIR R4 (v4.0.1) REST API enabling programmatic access to diagnostic imaging resources, AI-generated findings, and clinical workflow data. The FHIR API serves as the modern interoperability layer for integration with EHR systems, clinical decision support tools, population health platforms, and third-party applications. This document provides a complete reference for all supported FHIR resource types, RESTful interactions, search parameters, authentication mechanisms, and code examples. The API conforms to the FHIR US Core Implementation Guide (v6.1) and the IHE Radiology profiles (RAD-141, RAD-142) for radiology-specific workflows.

2. Authentication & Authorization

2.1 SMART on FHIR

Skolyn's FHIR API supports the SMART on FHIR authorization framework (SMART App Launch v2.0), enabling secure third-party application access with granular scope-based permissions. The authorization flow follows the standard OAuth 2.0 Authorization Code grant with PKCE:

# Discovery endpoint
GET /fhir/.well-known/smart-configuration

# Response
{
  "authorization_endpoint": "https://auth.skolyn.com/authorize",
  "token_endpoint": "https://auth.skolyn.com/token",
  "capabilities": ["launch-standalone", "launch-ehr", "client-public", "sso-openid-connect"],
  "scopes_supported": [
    "patient/DiagnosticReport.read",
    "patient/ImagingStudy.read",
    "patient/Observation.read",
    "system/DiagnosticReport.read",
    "system/*.read"
  ]
}

2.2 OAuth 2.0 Client Credentials

For backend system-to-system integrations, Skolyn supports the OAuth 2.0 Client Credentials grant. Client credentials are issued via the Management Console and scoped to specific FHIR resource types and operations. Tokens have a configurable lifetime (default 1 hour) and must be refreshed before expiration.

4. DiagnosticReport API

4.1 Read DiagnosticReport

GET /fhir/DiagnosticReport/dr-12345
Authorization: Bearer <token>
Accept: application/fhir+json

# Response (200 OK)

4.2 Search DiagnosticReports

# Search by patient and date range
GET /fhir/DiagnosticReport?patient=Patient/pat-67890&date=ge2026-01-01&date=le2026-02-28&_sort=-date&_count=20

# Search by status
GET /fhir/DiagnosticReport?status=final&category=LP29684-5

# Search by accession number
GET /fhir/DiagnosticReport?identifier=http://hospital.com/accession|ACC-2026-001

5. Observation API (AI Findings)

Each AI-generated finding is represented as a FHIR Observation resource linked to its parent DiagnosticReport. Observations include coded finding types (SNOMED CT, RadLex), numeric measurements with units (UCUM), confidence scores, and references to attention map Binary resources.

{
  "resourceType": "Observation",
  "id": "obs-001",
  "status": "final",
  "category": [{ "coding": [{ "system": "http://terminology.hl7.org/CodeSystem/observation-category", "code": "imaging" }] }],
  "code": { "coding": [{ "system": "http://snomed.info/sct", "code": "425771", "display": "Ring-enhancing lesion" }] },
  "subject": { "reference": "Patient/pat-67890" },
  "valueString": "Right temporal lobe enhancing lesion, 12x8mm",
  "component": [
    {
      "code": { "coding": [{ "system": "http://skolyn.com/fhir/cs/ai-metrics", "code": "confidence" }] },
      "valueQuantity": { "value": 0.94, "unit": "probability", "system": "http://unitsofmeasure.org" }
    },
    {
      "code": { "coding": [{ "system": "http://skolyn.com/fhir/cs/ai-metrics", "code": "attention-map" }] },
      "valueReference": { "reference": "Binary/attn-map-001" }
    }
  ]
}

6. Pagination & Bundle Responses

Search results are returned as FHIR Bundle resources with pagination support. The default page size is 20 resources, configurable up to 100 via the _count parameter. Navigation links (self, next, previous) are included in the Bundle for cursor-based pagination. Total result counts are returned in Bundle.total when the count is below 10,000; larger result sets return an estimate with an OperationOutcome warning.
