1. Executive Summary

Skolyn's security architecture is engineered on a Zero Trust, Defense-in-Depth model that enforces strict authentication, encryption, and access controls at every layer of the platform. As a medical imaging AI system processing Protected Health Information (PHI) and Personally Identifiable Information (PII), Skolyn adopts the highest standards mandated by HIPAA, GDPR, ISO 27001, and SOC 2 Type II frameworks. This document describes the complete security architecture, from network perimeter to application-layer protections, data encryption, identity and access management, incident response, and continuous monitoring.

Critical: All personnel handling PHI must complete Skolyn Security Awareness Training (SAT) annually. Non-compliance will result in access revocation within 24 hours.

2. Security Design Principles

Skolyn's security posture is governed by the following foundational principles, each validated through regular penetration testing and audit cycles:

  • Zero Trust Architecture (ZTA): No implicit trust is granted to any user, device, or network segment. Every request is authenticated, authorized, and encrypted regardless of origin. Network microsegmentation ensures east-west traffic is inspected.
  • Least Privilege Access: Every identity — human or service — is granted the minimum permissions required to perform its function. Permissions are reviewed quarterly and revoked automatically upon role change.
  • Defense in Depth: Multiple overlapping security controls ensure that a failure in one layer does not compromise the entire system. Controls span network, application, data, and physical layers.
  • Secure by Default: All platform components ship with hardened configurations. Debug endpoints, verbose logging, and administrative interfaces are disabled in production by default.
  • Assume Breach: The architecture is designed to detect, contain, and recover from compromises rapidly. Blast radius limitation, automated quarantine, and immutable audit trails enable effective incident response.

3.2 Perimeter Security

  • Web Application Firewall (WAF): Cloud-native WAF with OWASP Core Rule Set (CRS) v4.x, deployed in front of all public endpoints. Custom rules block SQL injection, XSS, path traversal, and medical-specific attack vectors.
  • DDoS Protection: Volumetric and application-layer DDoS mitigation with automatic traffic scrubbing. Capacity exceeds 10 Tbps of attack bandwidth absorption.
  • TLS Termination: All external connections terminate TLS 1.3 at the edge. Cipher suites are restricted to AEAD algorithms (AES-256-GCM, ChaCha20-Poly1305). TLS 1.0/1.1 are permanently disabled.

3.3 Service Mesh

All internal communication between microservices is secured using a service mesh (Istio/Linkerd) with mandatory mutual TLS (mTLS). Service-to-service authentication is enforced through SPIFFE identity certificates with a maximum validity of 24 hours, automatically rotated by the mesh control plane. Authorization policies define fine-grained access rules between services, ensuring that the inference engine cannot directly access the audit log service, for example.

4. Identity & Access Management (IAM)

4.1 Authentication

Skolyn supports enterprise-grade authentication mechanisms to integrate seamlessly with existing hospital identity infrastructure:

  • SAML 2.0 SSO: Integration with Microsoft Entra ID (Azure AD), Okta, Ping Identity, and other SAML-compliant IdPs. Assertion validation includes signature verification, audience restriction, and NotOnOrAfter enforcement.
  • OpenID Connect (OIDC): Support for OIDC flows (Authorization Code + PKCE) for web and mobile clients. ID tokens are validated against the IdP's JWKS endpoint.
  • Multi-Factor Authentication (MFA): Mandatory for all administrative access. Supported factors include TOTP (RFC 6238), FIDO2/WebAuthn hardware keys, and push notifications via authenticator apps.
  • Service Accounts: Machine-to-machine authentication uses short-lived JWT tokens issued by the platform's internal OAuth 2.0 authorization server. Token lifetime is capped at 15 minutes with automatic refresh.

4.2 Authorization Model

Access control follows a hybrid RBAC + ABAC model. Role-Based Access Control (RBAC) defines coarse-grained permissions tied to organizational roles (Radiologist, Administrator, Technician), while Attribute-Based Access Control (ABAC) enforces contextual policies such as time-of-day restrictions, IP geolocation, and device posture:

5. Data Protection

5.1 Encryption at Rest

All persistent data is encrypted using AES-256-GCM with envelope encryption. Data Encryption Keys (DEKs) are generated per-object and encrypted by a Key Encryption Key (KEK) stored in a FIPS 140-2 Level 3 certified Hardware Security Module (HSM). Key rotation occurs every 90 days automatically. Decrypted DEKs are held in memory only during active processing and are zeroized upon completion.

5.2 Encryption in Transit

All data in transit is protected by TLS 1.3. Internal service-to-service communication uses mTLS with SPIFFE-issued X.509 certificates. DICOM connections use DICOM TLS (compliant with DICOM PS3.15 Annex B) for imaging data transfer. HL7v2 messages are transmitted over TLS-encrypted MLLP channels.

5.3 PHI De-identification

For research and model training pipelines, Skolyn applies Safe Harbor or Expert Determination de-identification methods per HIPAA §164.514. Automated stripping removes 18 HIPAA identifiers from DICOM headers and replaces them with pseudonymized tokens. Pixel-level PHI (burned-in annotations) is detected and redacted using a dedicated OCR-based scrubber before any data exits the clinical environment.

6. Application Security

6.1 Secure Development Lifecycle (SDL)

All Skolyn code follows a rigorous Secure Development Lifecycle:

  • Static Analysis (SAST): Automated scanning on every pull request using CodeQL and Semgrep. Zero critical/high findings policy enforced — PRs with unresolved findings cannot be merged.
  • Dynamic Analysis (DAST): Weekly automated DAST scans against staging environments using OWASP ZAP. Full pen-test reports are generated monthly.
  • Software Composition Analysis (SCA): All third-party dependencies are scanned for known CVEs. Dependabot and Snyk alerts trigger automated patching workflows within 48 hours for critical vulnerabilities.
  • Container Security: Base images are rebuilt weekly from hardened, minimal distributions (distroless). Runtime scanning with Falco detects anomalous container behavior (unexpected process execution, file system modifications, network connections).

6.2 API Security

All API endpoints are protected by rate limiting (per-client, sliding window), request size limits (10 MB default, 512 MB for DICOM uploads), and input validation. API keys are hashed using Argon2id before storage. OAuth 2.0 scopes enforce fine-grained access to specific API resources. All API responses include security headers (HSTS, CSP, X-Content-Type-Options, X-Frame-Options).

7.2 Response Procedures

Skolyn maintains a documented Incident Response Plan (IRP) aligned with NIST SP 800-61 Rev. 2. The plan includes four phases: Preparation, Detection & Analysis, Containment & Eradication, and Post-Incident Activity. Automated containment actions include: revoking compromised credentials within 60 seconds, isolating affected pods via NetworkPolicy updates, capturing forensic snapshots of compromised containers, and triggering breach notification workflows to Data Controllers within 24 hours (per GDPR Art. 33).

8. Continuous Security Monitoring

Skolyn operates a 24/7 Security Operations Center (SOC) with the following capabilities:

  • SIEM Integration: All security events are forwarded to a centralized SIEM (Splunk/Elastic Security) for correlation, alerting, and forensic analysis. Log retention: 7 years for compliance.
  • Threat Intelligence: Real-time threat feeds from MITRE ATT&CK, CISA advisories, and healthcare-specific ISAC alerts are ingested and correlated against platform telemetry.
  • Vulnerability Management: Continuous vulnerability scanning with scheduled remediation SLAs (Critical: 24h, High: 7d, Medium: 30d, Low: 90d).
  • Penetration Testing: Annual third-party penetration tests by CREST-certified firms. Bug bounty program for responsible disclosure.
