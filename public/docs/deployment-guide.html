1. Executive Summary

This Deployment Guide defines the processes, automation, and governance that manage how Skolyn software progresses from development through staging to production. Given that Skolyn processes Protected Health Information (PHI) and provides AI-assisted diagnostic findings, deployment processes must be auditable, reversible, and validated at every stage. This document covers the CI/CD pipeline architecture, environment topology, release versioning, canary and blue-green deployment strategies, rollback procedures, database migration management, model registry promotion, and post-deployment verification. All deployments are automated through GitOps, with human approval gates at critical checkpoints to ensure patient safety and regulatory compliance.

2.1 Environment Isolation

Each environment runs in a dedicated Kubernetes namespace with strict network policy isolation. Production and DR environments are in separate cloud regions (or separate physical clusters for on-premises). Service mesh policies enforce that development environment services cannot communicate with staging or production services under any circumstances. Database credentials, encryption keys, and certificates are unique per environment — there is no shared secret material between environments.

3. CI/CD Pipeline Architecture

3.1 Pipeline Stages

The Skolyn CI/CD pipeline is implemented in GitHub Actions with the following stages:

  1. Code Quality (automated): Lint (Ruff, ESLint), type checking (mypy, tsc), formatting verification (Black, Prettier). Failure blocks all subsequent stages.
  2. Unit Tests (automated): Pytest suite with 85% minimum code coverage requirement. Tests run in parallel across a matrix of Python 3.11/3.12 and PyTorch 2.1/2.2 combinations.
  3. Security Scan (automated): CodeQL static analysis, Snyk dependency vulnerability scan, Trivy container image scan, and SBOM generation. Any critical or high vulnerability blocks the pipeline.
  4. Container Build (automated): Multi-stage Docker build producing minimal production images. Images are signed with Cosign and pushed to the authenticated container registry with content-addressable SHA256 tags.
  5. Integration Tests (automated): Deploy to Integration environment and run API contract tests, DICOM conformance tests, and HL7/FHIR interoperability tests. Tests use synthetic DICOM studies covering all 49 subtypes.
  6. Staging Deployment (automated with approval): Deploy to Staging via ArgoCD. Run clinical simulation tests with anonymized real studies. Human approval required from Clinical Validation and QA leads.
  7. Production Deployment (manual trigger): Canary deployment to 5% production traffic, monitored for 30 minutes. Progressive rollout to 25%, 50%, 100% with automated rollback if error rates exceed thresholds.

3.2 GitOps Workflow

All deployment configuration is stored in a dedicated skolyn-deploy Git repository. ArgoCD monitors this repository and reconciles the desired state (Git) with the actual state (Kubernetes) every 30 seconds. Changes to deployment manifests require peer review (minimum 2 approvals for production changes) and pass automated policy validation (OPA Gatekeeper admission controller) before merge.

4. Deployment Strategies

4.1 Canary Deployment (Default)

Canary deployment is the default strategy for all Skolyn service updates. The process works as follows:

  • Phase 1 (5% traffic, 30 min): New version receives 5% of incoming study processing requests. Automated monitoring tracks error rates, latency percentiles (p50, p95, p99), GPU utilization, and inference accuracy metrics against a baseline of the previous version.
  • Phase 2 (25% traffic, 60 min): If Phase 1 metrics are within acceptable bounds (error rate < baseline + 0.5%, p99 latency < baseline + 20%), traffic shifts to 25%. Continue monitoring with expanded metric collection including per-modality accuracy.
  • Phase 3 (50% traffic, 60 min): When Phase 2 clears, expand to 50%. Additional monitoring includes clinical finding concordance — AI findings from the canary version are compared against findings from the baseline version on the same studies (shadow mode analysis).
  • Phase 4 (100% traffic): Full rollout with 24-hour monitoring window before the old version is decommissioned.

4.2 Blue-Green Deployment (Major Versions)

For major version upgrades that include database schema changes or breaking API modifications, Skolyn uses blue-green deployment. Both the old (blue) and new (green) environments run simultaneously with full infrastructure parity. Traffic is switched at the load balancer level after comprehensive validation in the green environment. The blue environment remains available for instantaneous rollback for 72 hours after the switch.

5. AI Model Deployment

5.1 Model Registry

AI model artifacts are managed through the Skolyn Model Registry, which tracks model versions, training metadata, validation metrics, and deployment history. Each model version undergoes a formal promotion process:

  1. Research → Candidate: Model achieves target metrics on internal validation dataset. Automated comparison against current production model on benchmark suite.
  2. Candidate → Staging: Independent clinical validation team evaluates model on 1,000+ anonymized studies per subtype with blinded comparison against radiologist ground truth.
  3. Staging → Production: Regulatory approval (for regulated markets), clinical safety review sign-off, and deployment approval from the Clinical AI Safety Board.

5.2 Model Versioning

Models follow semantic versioning: MAJOR.MINOR.PATCH. Major versions indicate architecture changes requiring revalidation. Minor versions indicate training data updates or hyperparameter tuning. Patch versions indicate bug fixes or performance optimizations with no clinical impact.

7. Post-Deployment Verification

Every production deployment includes automated post-deployment verification (PDV) that runs a synthetic "golden study" through the complete pipeline — from DICOM ingestion to report delivery — for each enabled module. The golden study results are compared against known correct outputs. Any discrepancy triggers an automatic rollback. Additionally, production smoke tests verify DICOM connectivity, HL7 message routing, FHIR endpoint availability, and authentication flows.
