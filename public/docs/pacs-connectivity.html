1. Introduction

Skolyn integrates with institutional Picture Archiving and Communication Systems (PACS) as a transparent, inline or side-channel AI analysis node. The platform supports connectivity with all major PACS vendors including Sectra, Agfa Enterprise Imaging, Philips IntelliSpace, GE Centricity, Fujifilm Synapse, Hologic, and open-source solutions like Orthanc and DCM4CHEE. This document provides vendor-specific configuration guides, routing rule templates, and troubleshooting procedures to enable seamless PACS integration. Whether deployed as a DICOM routing destination, a query-retrieve source, or a DICOMweb client, Skolyn adapts to your existing imaging infrastructure without requiring workflow modifications for radiologists or technologists.

2. Integration Architectures

2.1 Inline (Router) Architecture

In the inline architecture, studies flow from the modality through Skolyn before reaching the PACS. The DICOM router (typically an interface engine like Laurel Bridge Compass or Mirth Connect) forwards a copy of each incoming study to Skolyn's DICOM Gateway while simultaneously forwarding the original to the PACS. Skolyn processes the study and sends the AI results back as a new series (DICOM SR, secondary capture key images) to the PACS under the same Study Instance UID. This architecture ensures AI results appear alongside source images in the radiologist's reading workflow.

2.2 Side-Channel Architecture

In the side-channel architecture, Skolyn operates as a DICOM Query/Retrieve SCU, pulling completed studies from the PACS on a configurable schedule or triggered by HL7 ORM messages from the RIS. This approach requires no changes to the existing DICOM routing infrastructure and is ideal for sites that cannot modify their modality-to-PACS routing. Skolyn queries the PACS for new studies matching configured criteria (modality, body part, study description), retrieves the instances via C-MOVE or C-GET, processes them, and sends results back to the PACS via C-STORE.

2.3 Hybrid Architecture

The hybrid architecture combines inline forwarding for high-priority modalities (CT, MR for stroke/trauma) with side-channel retrieval for lower-priority workloads (routine X-ray, mammography screening). This provides sub-minute turnaround for STAT studies while minimizing network bandwidth impact for the bulk of the imaging volume.

4. Result Delivery to PACS

4.1 DICOM Structured Report

Skolyn generates DICOM SR (TID 1500 Measurement Report) for each analyzed study. The SR is stored as a new series within the original study, with series description "Skolyn AI Analysis" and series number 999. The SR contains coded findings using SNOMED CT, RadLex, and LOINC terminologies, numeric measurements with UCUM units, and references to attention map key images. Most modern PACS viewers can display DICOM SR content inline or in a dedicated panel.

4.2 Key Images (Secondary Capture)

For PACS systems that have limited SR display capabilities, Skolyn generates secondary capture images with AI overlay annotations (bounding boxes, attention heatmaps, measurement callouts) burned into the pixel data. These key images are stored as a separate series within the study, providing a visual summary of AI findings that is viewable on any DICOM-compliant workstation.

6. Troubleshooting

  • Association rejected: Verify AE Title is registered in both Skolyn and PACS. Check that the calling AE Title matches the registered entry exactly (case-sensitive).
  • Transfer syntax negotiation failure: Ensure at least one common transfer syntax is configured. Use Explicit VR Little Endian as the universal fallback.
  • Studies not arriving: Verify network connectivity (echoscu test), firewall rules (port 11112 inbound), and TLS certificate validity.
  • Results not appearing in PACS: Verify the return C-STORE association from Skolyn to PACS is configured. Check that the Study Instance UID matches the original study.
