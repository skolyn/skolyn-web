Worklist Management Guide

Skolyn Documentation - Version 2.0 - Last updated: February 2026

1. Overview

Skolyn's Worklist Management system orchestrates study processing, prioritization, and status tracking across all imaging modules. The worklist provides a centralized view of all studies flowing through the platform, from initial reception through AI analysis to result delivery. It enables intelligent prioritization based on clinical urgency, AI confidence levels, and institutional routing rules.

The worklist integrates with DICOM Modality Worklist (MWL), HL7v2 ordering, and FHIR ServiceRequest to provide a unified scheduling and tracking interface. Administrators can monitor processing capacity, identify bottlenecks, and configure routing rules through the Management Console.

2. DICOM Modality Worklist

2.1 MWL SCP Configuration

Skolyn operates as a Modality Worklist SCP (Service Class Provider) to provide scheduled study information to imaging modalities and PACS systems.

# skolyn.yaml - MWL configuration
worklist:
  mwl:
    enabled: true
    port: 11113
    aeTitle: SKOLYN_MWL
    matchingKeys:
      - PatientID
      - AccessionNumber
      - ScheduledProcedureStepStartDate
      - Modality
      - ScheduledStationAETitle
    returnKeys:
      - PatientName
      - PatientID
      - PatientBirthDate
      - PatientSex
      - AccessionNumber
      - RequestedProcedureDescription
      - ScheduledProcedureStepDescription
      - Modality
      - ScheduledStationAETitle
      - ScheduledProcedureStepStartDate
      - ScheduledProcedureStepStartTime

3.2 Auto-Prioritization Rules

Skolyn automatically adjusts study priority based on clinical context and AI triage results:

# skolyn.yaml - Auto-prioritization rules
worklist:
  autoPriority:
    rules:
      - name: "ED Studies STAT"
        condition: "study.patientLocation MATCHES 'ED|ER|EMERGENCY'"
        priority: STAT
        
      - name: "ICU Studies Urgent"
        condition: "study.patientLocation MATCHES 'ICU|CCU|NICU'"
        priority: Urgent
        
      - name: "Stroke Protocol STAT"
        condition: "study.protocolName MATCHES 'STROKE|CTA HEAD'"
        priority: STAT
        
      - name: "Screening Mammography Low"
        condition: "study.modality == 'MG' AND study.indication MATCHES 'SCREENING'"
        priority: Low
        
      - name: "AI Triage Upgrade"
        condition: "study.triageResult.criticalFinding == true"
        priority: STAT
        description: "Upgrade to STAT when AI triage detects critical finding"

4. Routing Rules

4.1 Module Routing

Routing rules determine which AI module(s) process each incoming study. Rules are evaluated in order; the first matching rule determines the routing.

# skolyn.yaml - Module routing rules
worklist:
  routing:
    rules:
      - name: "MRI to Rhenium"
        condition: "study.modality == 'MR'"
        module: rhenium
        
      - name: "CT to Seaborgium"
        condition: "study.modality == 'CT'"
        module: seaborgium
        
      - name: "Ultrasound to Scandium"
        condition: "study.modality == 'US'"
        module: scandium
        
      - name: "X-Ray to Terbium"
        condition: "study.modality IN ['CR', 'DX', 'MG']"
        module: terbium
        
      - name: "Multi-modality Study"
        condition: "study.modalitiesInStudy.size() > 1"
        module: auto  # Analyze with all relevant modules

7. Multi-Tenant Worklist

In multi-tenant deployments (cloud SaaS), each institution has an isolated worklist with dedicated queue capacity and independent routing rules. Fair-share scheduling ensures that no single institution can monopolize processing resources. Administrators can configure per-institution quotas and burst capacity.

SettingDescriptionDefault
Guaranteed ThroughputMinimum studies/hour per institution50
Burst CapacityMaximum studies/hour during peak200
Queue Depth LimitMaximum pending studies per institution500

8. Retry and Error Handling

# skolyn.yaml - Retry configuration
worklist:
  retry:
    maxRetries: 3
    initialDelay: 60
    maxDelay: 900
    backoffMultiplier: 3
    retryableErrors:
      - GPU_OUT_OF_MEMORY
      - TIMEOUT
      - TEMPORARY_FAILURE
    nonRetryableErrors:
      - INVALID_DICOM
      - UNSUPPORTED_MODALITY
      - LICENSE_EXCEEDED
