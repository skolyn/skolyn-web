1. Executive Summary

This installation guide provides comprehensive step-by-step instructions for deploying the Skolyn Diagnostic AI Platform in on-premises and hybrid environments. The guide covers hardware prerequisites, operating system preparation, Kubernetes cluster setup, GPU driver installation, Skolyn Helm chart deployment, database migration, module activation, and post-installation verification. For cloud-hosted (SaaS) deployments, no installation is required — refer to the Quick Start Guide (SKL-GS-002) for initial configuration. All on-premises deployments require a valid Skolyn Enterprise license key and must be performed by authorized Skolyn Deployment Engineers or certified partner personnel.

Important: On-premises installation requires coordination with the Skolyn Deployment Engineering team. Contact your Skolyn Account Manager to schedule an installation engagement before proceeding.

2.2 Capacity Planning

The hardware specifications above are sized based on study processing volume. The minimum configuration supports up to 500 studies per hour, the recommended configuration supports up to 2,500 studies per hour, and the high-volume configuration supports up to 12,000 studies per hour. GPU memory is the primary bottleneck for MRI multi-sequence analysis — each concurrent MRI study requires approximately 6 GB of GPU memory during inference. Capacity planning worksheets are available from your Skolyn Account Manager.

3. Software Prerequisites

3.1 Operating System

  • Supported OS: Ubuntu Server 22.04 LTS, RHEL 8.8+, RHEL 9.x, SUSE Linux Enterprise Server 15 SP5+
  • Kernel: Version 5.15+ (required for GPU passthrough and container runtime)
  • Container Runtime: containerd 1.7+ (Docker Engine is supported but not recommended for production)
  • Kubernetes: v1.28+ (supported distributions: kubeadm, RKE2, EKS-Anywhere, OpenShift 4.14+)

4. Kubernetes Cluster Setup

4.1 Cluster Architecture

The Skolyn deployment requires a minimum 3-node Kubernetes cluster: one control plane node and two worker nodes (at least one with GPUs). For production deployments, we recommend 3 control plane nodes (for HA) and a minimum of 3 worker nodes with GPUs.

4.2 GPU Operator Installation

# Install NVIDIA GPU Operator
helm repo add nvidia https://helm.ngc.nvidia.com/nvidia
helm install gpu-operator nvidia/gpu-operator \
  --namespace gpu-operator --create-namespace \
  --set driver.enabled=false \
  --set toolkit.enabled=true \
  --set devicePlugin.enabled=true \
  --set mig.strategy=mixed

# Verify GPU nodes
kubectl get nodes -o json | jq '.items[].status.allocatable["nvidia.com/gpu"]'

4.3 Storage Configuration

Skolyn requires two storage classes: a high-performance class for database and inference cache (minimum 3,000 IOPS), and a capacity-optimized class for DICOM object storage and backup archives. Supported CSI drivers include Longhorn, Rook-Ceph, OpenEBS, and vendor-specific CSI drivers (Dell, NetApp, Pure Storage).

5. Skolyn Platform Installation

5.1 Helm Repository Setup

# Add Skolyn Helm repository
helm repo add skolyn https://charts.skolyn.com --username license --password <LICENSE_KEY>
helm repo update

# Verify chart availability
helm search repo skolyn/skolyn-platform --versions

5.2 Configuration Values

Create a values.yaml file with your deployment-specific settings. The configuration covers database credentials, storage class references, GPU resource requests, TLS certificate paths, DICOM AE Titles, and module activation flags. A sample values file is included in the Helm chart documentation:

# skolyn-values.yaml
global:
  licenseKey: "SKL-ENT-XXXX-XXXX-XXXX"
  domain: "skolyn.your-hospital.com"
  tls:
    enabled: true
    certManager: true  # Use cert-manager for automatic TLS

database:
  type: postgresql
  host: "postgresql.skolyn.svc"
  port: 5432
  name: skolyn_prod
  credentials:
    existingSecret: skolyn-db-credentials

storage:
  dicom:
    storageClass: "longhorn-capacity"
    size: "5Ti"
  cache:
    storageClass: "longhorn-performance"
    size: "500Gi"

inference:
  gpu:
    request: 2
    type: "nvidia.com/gpu"
  replicas: 2
  autoscaling:
    enabled: true
    minReplicas: 2
    maxReplicas: 8
    targetGPUUtilization: 70

modules:
  rhenium: { enabled: true }
  seaborgium: { enabled: true }
  scandium: { enabled: true }
  terbium: { enabled: true }

5.3 Deploy

# Create namespace
kubectl create namespace skolyn

# Install Skolyn
helm install skolyn skolyn/skolyn-platform \
  --namespace skolyn \
  --values skolyn-values.yaml \
  --timeout 15m \
  --wait

# Verify all pods are running
kubectl get pods -n skolyn -w

7. Upgrade Procedures

Platform upgrades are performed via Helm chart version bumps. Skolyn supports rolling upgrades with zero downtime for minor versions and scheduled maintenance windows for major versions. Always review the release notes and run the pre-upgrade compatibility check before upgrading:

# Check compatibility
helm diff upgrade skolyn skolyn/skolyn-platform --namespace skolyn --values skolyn-values.yaml

# Perform upgrade
helm upgrade skolyn skolyn/skolyn-platform --namespace skolyn --values skolyn-values.yaml --wait
