1. Executive Summary

Skolyn implements a Zero Trust identity and access management framework that enforces the principle of least privilege across all platform interactions. The system combines Role-Based Access Control (RBAC) with Attribute-Based Access Control (ABAC) to provide fine-grained, context-aware authorization decisions. This document defines the access control model, role hierarchy, permission granularity, session management policies, emergency access procedures, and integration patterns with enterprise identity providers. Every access decision is logged, auditable, and reversible to maintain full regulatory compliance with HIPAA, GDPR, and SOC 2 requirements.

2. Access Control Model

2.1 Hybrid RBAC + ABAC Architecture

Skolyn's access control architecture employs a two-layer authorization model. The first layer, Role-Based Access Control (RBAC), assigns permissions based on organizational roles. Each user is assigned one or more roles that define a baseline set of allowed actions. The second layer, Attribute-Based Access Control (ABAC), evaluates contextual attributes such as the user's location, device posture, time of day, and the sensitivity of the requested resource to make dynamic authorization decisions. This hybrid approach ensures that a Radiologist can view studies (RBAC) but only from compliant devices within the hospital network during working hours (ABAC).

2.2 Policy Decision Point (PDP)

All authorization requests are evaluated by a centralized Policy Decision Point implemented as an Open Policy Agent (OPA) sidecar deployed alongside each microservice. Policies are written in Rego and version-controlled in Git. Policy updates are deployed via GitOps with mandatory peer review and automated testing against a comprehensive decision matrix before production rollout.

3.2 Custom Roles

Institution Administrators can create custom roles by combining individual permissions from the platform's permission catalog. Custom roles allow granular delegation such as creating a "Trainee Radiologist" role that permits study viewing but requires attending co-signature for report finalization, or a "Research Coordinator" role limited to de-identified data export. All custom role definitions are audited and require documented justification.

4. Authentication Requirements

4.1 Identity Provider Integration

Skolyn acts as a Service Provider (SP) and delegates primary authentication to the institution's Identity Provider (IdP) via industry-standard protocols:

  • SAML 2.0: For enterprise SSO integration with Microsoft Entra ID, Okta, Ping Identity, and ADFS. Assertion attributes map directly to Skolyn roles and institutional affiliation.
  • OpenID Connect: Authorization Code flow with PKCE for web and mobile clients. ID tokens contain role claims that drive RBAC assignments.
  • LDAP/Active Directory: Direct bind authentication for on-premise deployments with group-to-role mapping synchronized every 5 minutes.

4.2 Multi-Factor Authentication

MFA is mandatory for all administrative roles and configurable per institution for clinical roles. Supported second factors include:

  • FIDO2/WebAuthn: Hardware security keys (YubiKey, Titan) — recommended for highest assurance.
  • TOTP: Time-based one-time passwords via Google Authenticator, Authy, or Microsoft Authenticator (RFC 6238).
  • Push Notifications: App-based push approval for mobile-friendly workflows.
  • Smart Cards: PIV/CAC support for government and military healthcare deployments.

Sessions are bound to the originating IP address and device fingerprint. Any change in these attributes triggers a re-authentication challenge. Session tokens are stored in HttpOnly, Secure, SameSite=Strict cookies with AES-256-GCM encryption.

6. Emergency Access (Break-Glass)

For life-threatening clinical emergencies where the assigned radiologist is unavailable, Skolyn provides a documented Break-Glass procedure:

  1. Initiation: The requesting clinician invokes the emergency access flow from the study viewer, providing a clinical justification (free text + structured reason code).
  2. Authorization: A second authorized user (Charge Radiologist, Department Head, or Administrator) must confirm the emergency access request in real-time via push notification.
  3. Access Grant: Temporary elevated access is granted for a maximum of 4 hours, limited to the specific study/patient context.
  4. Audit & Review: All break-glass events generate P2 security alerts and are flagged for mandatory review by the Compliance Officer within 24 hours. The clinical justification is preserved in the immutable audit trail.
Warning: Break-glass events that cannot be clinically justified after review may result in disciplinary action and are reported to the institution's Privacy Officer.

7. API Access Control

7.1 OAuth 2.0 Scopes

API access is governed by OAuth 2.0 scopes that map to specific platform capabilities:

# Read-only access to studies
scope: studies:read

# Write access to reports
scope: reports:write

# Administrative configuration
scope: admin:config

# Audit log access
scope: audit:read

# AI model management
scope: models:manage

7.2 Rate Limiting

API rate limits are enforced per-client using a sliding window algorithm. Default limits are 1,000 requests per minute for read operations and 200 requests per minute for write operations. Burst capacity allows 50% overage for 30-second windows. Rate limit exhaustion returns HTTP 429 with a Retry-After header indicating the reset time.

8. Access Reviews & Certification

Quarterly access reviews (User Access Reviews / UAR) are conducted by Institution Administrators with Security team oversight. The review process includes certification that each user's role and permissions remain appropriate for their current job function, identification and removal of dormant accounts (no login for 90 days), verification of service account permissions and ownership, and documentation of any exceptions with risk acceptance sign-off.
